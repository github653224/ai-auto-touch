# 视频流黑屏问题 - 最终修复方案

## 问题描述
在页面切换（如从设备列表到屏幕显示，或在不同菜单间切换）时，视频流会出现黑屏。一旦黑屏，所有页面的视频都无法恢复，必须重启后端才能正常使用。

## 根本原因

之前采用的是**多客户端共享一个 Streamer** 的架构，存在以下问题：
1. **状态共享导致状态污染**：当一个客户端出错时，会影响共享的 Streamer
2. **延迟停止时间不够**：React Router 切换页面时，组件卸载和挂载之间的时间间隔不固定
3. **错误状态无法恢复**：一旦 Streamer 进入错误状态，所有客户端都会黑屏
4. **复杂的状态管理**：需要维护客户端列表、引用计数、广播逻辑等

## 最终解决方案

参考 **AutoGLM-GUI-main** 项目的稳定实现，采用**一个 Socket 一个 Streamer** 的简单架构。

### 核心改变

1. **每个 Socket 连接独立的 Streamer**
   - 每个客户端连接时创建新的 Streamer
   - 客户端断开时立即清理对应的 Streamer
   - 不再共享 Streamer

2. **设备级别的锁机制**
   - 防止同时启动多个 scrcpy 服务器（会导致端口冲突）
   - 新连接会停止同一设备的旧连接

3. **立即清理，无延迟**
   - Socket 断开时立即停止对应的 Streamer
   - 不再使用延迟停止机制
   - 避免资源泄漏

### 架构对比

#### 旧架构（有问题）
```
设备 A
  └── Streamer A (共享)
       ├── 客户端 1 (Socket 1)
       ├── 客户端 2 (Socket 2)
       └── 客户端 3 (Socket 3)
```
- 问题：一个客户端出错，所有客户端都受影响

#### 新架构（稳定）
```
设备 A
  ├── Socket 1 → Streamer 1 (独立)
  ├── Socket 2 → Streamer 2 (独立)
  └── Socket 3 → Streamer 3 (独立)
```
- 优点：每个连接独立，互不影响

### 关键代码变化

#### 1. 数据结构简化
```python
# 旧：按设备分组
_device_streamers: dict[str, ScrcpyStreamer] = {}  # Key: device_id
_device_clients: dict[str, set[str]] = {}  # Key: device_id, Value: set of sids

# 新：按 Socket 分组
_socket_streamers: dict[str, ScrcpyStreamer] = {}  # Key: sid
_stream_tasks: dict[str, asyncio.Task] = {}  # Key: sid
```

#### 2. 连接逻辑简化
```python
# 旧：检查是否有现有流，尝试复用
if existing_streamer:
    # 加入客户端列表，发送元数据
    _device_clients[device_id].add(sid)
    # ...
else:
    # 创建新流
    # ...

# 新：每次都创建新流
async with device_lock:
    # 停止同设备的其他流
    for s in sids_to_stop:
        await _stop_stream_for_sid(s)
    
    # 创建新的 streamer
    streamer = ScrcpyStreamer(...)
    _socket_streamers[sid] = streamer
```

#### 3. 断开逻辑简化
```python
# 旧：引用计数 + 延迟停止
async def disconnect(sid: str):
    clients.discard(sid)
    if len(clients) == 0:
        asyncio.create_task(_delayed_stream_stop(device_id, 2))

# 新：立即停止
async def disconnect(sid: str):
    await _stop_stream_for_sid(sid)
```

## 优势

1. **简单可靠**：架构简单，逻辑清晰，易于维护
2. **隔离性好**：每个连接独立，互不影响
3. **无状态污染**：新连接总是获得全新的 Streamer
4. **自动恢复**：页面刷新或重新进入时自动创建新连接
5. **无需重启后端**：黑屏后刷新页面即可恢复

## 权衡

**资源消耗**：
- 旧架构：多个客户端共享一个 scrcpy 进程
- 新架构：每个客户端一个 scrcpy 进程

**实际影响**：
- 对于单用户场景（我们的主要使用场景），资源消耗几乎相同
- 对于多用户场景，新架构会消耗更多资源，但换来了稳定性
- scrcpy 进程本身很轻量，多开几个进程的影响可以接受
